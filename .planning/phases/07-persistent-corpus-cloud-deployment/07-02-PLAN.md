    Create `Dockerfile` (project root) as a single-stage image per D-07/D-08:
    - Base `FROM python:3.12-slim` (NOT alpine — psycopg[binary] ships glibc manylinux wheels that do not run on musl).
    - `WORKDIR /app`.
    - `COPY requirements.txt ./` then `RUN pip install --no-cache-dir -r requirements.txt` (requirements-first ordering for layer-cache efficiency). No `apt-get` line is needed — psycopg[binary] and yt-dlp[default] both install from wheels with no build toolchain.
    - `COPY app/ ./app/`, `COPY scripts/ ./scripts/`, `COPY data/ ./data/`. Do NOT copy `raw_data/` (the processed corpus lives in Railway Postgres, not the image filesystem) — Pitfall 7.
    - `RUN chmod +x scripts/start.sh`.
    - `CMD ["scripts/start.sh"]` (exec form invoking the shell script). Do NOT write `CMD ["uvicorn", ..., "$PORT"]` — shell variables do not expand inside JSON-array exec form (Pattern 3 gotcha).

    Create `scripts/start.sh` per D-09:
    - Shebang `#!/usr/bin/env bash`.
    - `set -e` (a failed psql init must abort startup so Railway marks the deploy failed).
    - `psql "$DATABASE_URL" -f scripts/init_db.sql` (runs the already-idempotent DDL on every start).
    - `exec uvicorn app.main:app --host 0.0.0.0 --port "${PORT:-8000}"` — the `exec` replaces the bash process so uvicorn receives SIGTERM directly; `${PORT:-8000}` accepts Railway's injected PORT while preserving local `docker run` on 8000.
    Do NOT hardcode any secret value (DATABASE_URL, OPENAI_API_KEY, ANTHROPIC_API_KEY) in either file — all secrets arrive at runtime via Railway env injection (D-11).
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo → version control | Files committed to git must never contain secrets or corpus dumps |
| Railway env injection → container | Secrets enter the running container as process env, never from the image |
| Vercel build → browser bundle | NEXT_PUBLIC_API_URL is inlined into client JS at build time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-02 | Information Disclosure | Dockerfile / start.sh / committed config | mitigate | No secret literals in any file; secrets supplied via Railway env vars only (D-11). Verified by `git grep -nE "sk-ant\|sk-proj"` over Dockerfile and scripts/start.sh |
| T-07-03 | Information Disclosure | corpus_dump.sql | mitigate | Add `corpus_dump.sql` to `.gitignore` before any pg_dump runs (Plan 03), preventing the full corpus text from being committed |
| T-07-04 | Tampering | scripts/start.sh schema init | accept | init_db.sql is idempotent (IF NOT EXISTS throughout) and runs against the operator-controlled DATABASE_URL; `set -e` aborts on failure so a bad init never silently serves a broken DB |
| T-07-05 | Information Disclosure | NEXT_PUBLIC_API_URL in browser bundle | accept | The Railway service URL is intended to be public (HTTPS endpoint); exposing it in the client bundle is acceptable. The Vercel server-side rewrite still hides it from direct browser requests on the same-origin /api/py path (D-04) |
</threat_model>
