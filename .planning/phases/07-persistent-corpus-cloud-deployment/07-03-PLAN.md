<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → Railway/Vercel dashboards | Secrets are entered into provider env-var UIs, never into the repo |
| local Docker Postgres → Railway Postgres | corpus_dump.sql crosses the network over an authenticated DATABASE_PUBLIC_URL |
| public internet → Railway/Vercel HTTPS | Untrusted visitors reach the app over TLS-terminated public endpoints |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Information Disclosure | API keys during deploy | mitigate | Keys entered only in Railway dashboard env vars (D-11); verified by `git grep -rn "sk-ant\|sk-proj"` returning nothing across the repo |
| T-07-07 | Information Disclosure | corpus_dump.sql artifact | mitigate | Gitignored (Plan 02) and deleted after the confirmed Railway restore (Task 3 step 9) |
| T-07-08 | Elevation of Privilege | Railway Postgres external access | accept | Railway exposes the DB only via an authenticated DATABASE_PUBLIC_URL TCP proxy; no anonymous access (RESEARCH.md Security Domain) |
| T-07-09 | Spoofing/Tampering | public HTTPS endpoints | accept | Railway and Vercel auto-provision and manage TLS certificates; no hand-rolled TLS (D-03) |
| T-07-10 | Information Disclosure | Railway URL in browser bundle | accept | NEXT_PUBLIC_API_URL is intentionally a public HTTPS endpoint; the Vercel server-side rewrite keeps client requests same-origin on /api/py (D-04) |
</threat_model>
