Chunking algorithm (D-01, D-02 — implement exactly):

1. Treat each `.txt` file as one document.
2. Split on blank lines (`\n\n+`) to get paragraph blocks. Files are already flat blank-line-separated paragraphs (D-02 — verified, no `>` or `[quote]` markers in the corpus).
3. Token-count each paragraph with `tiktoken.get_encoding("cl100k_base")`.
4. Greedy pack: accumulate paragraphs into a current chunk; close the chunk when adding the next paragraph would push it over 500 tokens, then start a new chunk.
5. Forward-merge: if a paragraph is < 40 words (whitespace-split word count), do NOT close the chunk before it — merge it forward into the next chunk rather than indexing a sub-40-word standalone chunk.
6. If the entire document is < 300 tokens, emit one chunk anyway (don't drop short documents).
7. `chunk_index` is 0-based and resets per document (D-04).
8. Compute `content_hash = sha256(chunk.text.encode("utf-8")).hexdigest()`.
9. `metadata` MUST include `{"source_filename": raw_doc.source_id}` (D-03 — needed by Plan 05 eval-authoring UI).

Normalization (applied to both loader output and pre-hash text):

- `unicodedata.normalize("NFKC", raw)` — collapses ligatures and width variants.
- Strip leading/trailing whitespace on the whole document.
- Preserve internal blank-line paragraph separators (DO NOT collapse `\n\n` to space — the chunker needs them).

Token budget: target 300–500 tokens per chunk; treat 500 as a hard cap, 300 as a soft floor (the floor is informational — do not pad chunks). Files in `raw_data/forum_posts/` are ~500–1300 tokens each, so most files produce 1–3 chunks.
