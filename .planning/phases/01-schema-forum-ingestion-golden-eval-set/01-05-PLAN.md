<objective>
Author the golden eval set for retrieval evaluation. After Plans 01–04 the forum corpus is in Postgres as embedded chunks. This plan builds the scripted helper `python -m app.eval.author` (D-07), uses it to interactively produce `eval/golden_set.jsonl` with at least 20 `(query, expected_chunk_ids, expected_themes, held_out)` tuples (EVAL-01), and locks the held-out subset (5 tuples) in `eval/HELD_OUT.md` with a timestamp BEFORE any Phase 2 retrieval tuning occurs (D-10, D-11).

Purpose: Phase 5 measures recall@K and MRR against this eval set. If even one retrieval parameter is touched before the held-out split is locked, the eval is contaminated — any "improvement" could be tuning-to-the-set. Locking this set in Phase 1 is the hard ordering constraint that makes Phase 5's numbers meaningful.

Output: A schema module + an interactive CLI + the artifacts the CLI produces (`eval/QUERIES.md` as input, `eval/golden_set.jsonl` and `eval/HELD_OUT.md` as output). Includes a `checkpoint:human-action` task for the irreducibly-human accept/reject loop.

<!-- The locked JSONL schema and helper signatures. -->

JSONL schema (D-08), one JSON object per line:

```json
{
  "query": "What amp settings does BB King use for a clean blues tone?",
  "expected_chunk_ids": ["b8c7d6e5-...", "9a1b2c3d-..."],
  "expected_themes": ["amp_settings"],
  "held_out": false
}
```

Field rules:
- `query`: non-empty string, max 300 chars.
- `expected_chunk_ids`: non-empty list of UUID strings; each UUID MUST exist in `chunks.id` at authoring time. The authoring script verifies this by selecting only from live DB results.
- `expected_themes`: non-empty list whose every element is in `VALID_THEMES = ("amp_settings", "pedal_choice", "signal_chain", "pickup_tone", "studio_vs_live")` (D-09 fixed enum).
- `held_out`: boolean.

Held-out manifest (D-10, D-11) — `eval/HELD_OUT.md` template:

```
# Held-Out Golden Eval Set

**Locked at:** 2026-05-15T18:42:11Z  (ISO-8601 UTC)
**Total tuples:** 20
**Held-out indices (0-based, matching golden_set.jsonl line order):** [2, 7, 11, 14, 18]
**Statement:** No Phase 2 retrieval tuning has been performed at the time of this commit.
                Recall@K and MRR (Phase 5) will be measured against these five held-out indices,
                which are sourced from five distinct forum topics:
                - bb_king_tone.txt
                - john_mayer_tone.txt
                - funk_tone.txt
                - eddie_van_halen_tone.txt
                - modern_pop_punk_tone.txt

**Audit:** This file MUST be committed to git BEFORE any Phase 2 retrieval parameter
           (K, expansion strategy, alias map content) is touched.
```

Schema helper signature (`app/eval/schema.py`):

- `VALID_THEMES: tuple[str, ...]` — exact tuple `("amp_settings","pedal_choice","signal_chain","pickup_tone","studio_vs_live")`.
- `Theme = Literal[...]` — same five literal strings.
- `class GoldenTuple(BaseModel)` — pydantic v2 model. Fields: `query: str = Field(min_length=1, max_length=300)`, `expected_chunk_ids: list[str] = Field(min_length=1)`, `expected_themes: list[Theme] = Field(min_length=1)`, `held_out: bool`. `@field_validator("expected_chunk_ids")` parses each entry as `uuid.UUID(s)` and raises on failure.
- `def load_golden_set(path: Path) -> list[GoldenTuple]` — read JSONL line-by-line; raise `ValueError(f"Malformed JSONL at line {n}: {orig_error}")` on any parse/validation failure.
- `def save_golden_set(tuples: list[GoldenTuple], path: Path) -> None` — write JSONL preserving order; one `json.dumps(t.model_dump())` per line; newline-terminated.

CLI flow for `app/eval/author.py`:

1. Read draft queries from `eval/QUERIES.md` (one query per line; lines starting with `#` and blank lines ignored).
2. Construct `embedder = get_embedder()` and open `get_conn()`.
3. For each draft query:
   - Compute `q_vec = embedder.embed_query(query)`.
   - `SELECT c.id::text, c.chunk_text, c.metadata_json, d.source_id FROM chunks c JOIN documents d ON d.id = c.document_id ORDER BY c.embedding <=> %s::vector LIMIT %s` (parameterized).
   - Print each candidate with its 1-based index, source filename, and the first 200 chars of `chunk_text`.
   - Prompt: `Accept which chunks for this query? (e.g. "1,3,4" or "skip"): `.
   - If `skip`, do not record a tuple; continue.
   - Else parse the comma-separated indices; collect their UUIDs as `expected_chunk_ids`.
   - Prompt: `Themes (comma-separated, choices: amp_settings, pedal_choice, signal_chain, pickup_tone, studio_vs_live): `; parse; validate against the enum.
   - Prompt: `Held out? (y/N): `; default N.
   - Construct `GoldenTuple`; append to running list; on validation failure re-prompt.
4. After all queries reviewed, write `eval/golden_set.jsonl` via `save_golden_set`.
5. Compute held-out indices (those with `held_out=true`); write `eval/HELD_OUT.md` with the timestamp, indices, list of source forum topics covered, and the no-tuning statement.
6. Print a final summary: total tuples, held-out count, distribution by source filename.

Constraints (CLAUDE.md):
- The authoring script MUST go through `get_embedder()` — NEVER `import openai` directly.
- All SQL via `%s` parameterization.
- Chunk IDs read live from the DB (D-07: "no manual transcription").
